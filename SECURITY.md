# 安全策略

## 支持范围

当前安全维护范围为 `main` 分支的最新代码。正式发布版本后，将在本文件中补充受支持版本矩阵。

## 漏洞报告

如果你发现安全漏洞，请不要在公开 Issue 中披露可复现细节、真实凭据、生产地址或攻击载荷。请通过 GitHub Private Vulnerability Reporting 或维护者公开的安全联系方式私下报告。

报告时建议包含：

- 受影响的模块、接口或部署方式。
- 复现步骤和必要的最小请求示例。
- 影响评估，例如越权、信息泄露、XSS、SSRF、供应链风险等。
- 已知缓解方式或修复建议。

## 响应流程

- 维护者收到报告后会先确认影响范围。
- 对确认的漏洞，优先发布修复并补充测试。
- 在修复可用前，维护者可能先发布临时缓解建议。
- 漏洞公开披露应在修复发布后进行，避免提前扩大攻击面。

## 开发安全基线

提交前应执行：

```bash
npm run open-source:check
```

安全相关变更必须说明：

- 是否改变认证、授权、CORS、Cookie、CSRF、安全响应头或日志脱敏行为。
- 是否新增外部依赖、Docker 镜像、GitHub Actions 权限或第三方服务。
- 是否已更新必要文档和测试。
